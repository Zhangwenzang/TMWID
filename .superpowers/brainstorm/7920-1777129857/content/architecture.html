三种架构方案

核心差异：Hook 如何写入状态文件，以及气泡应用如何读取

A

直接文件写入（推荐）

Hook 直接 echo JSON 到状态文件，SwiftUI 用 DispatchSourceFileSystemObject 监听。零依赖，最简单。

Pros

  • Hook 只需一行 shell，无需额外工具
  • SwiftUI 原生支持文件系统监听
  • 无中间进程，无额外延迟

Cons

  • 多窗口并发写入需处理竞态（用追加模式 + 原子写入）
  • 文件损坏时需要恢复
B

CLI 工具 + 状态目录

提供一个小型 CLI（Swift/Python），Hook 调用它来写入状态。CLI 负责原子写入、唯一 ID、错误处理。气泡应用读取结构化的状态目录。

Pros

  • 原子写入和错误处理封装在 CLI 里
  • 支持更多元数据（任务ID、窗口标识等）
  • 状态目录结构清晰，易于调试

Cons

  • 多一个编译产物需要安装
  • Hook 配置多了一步调用
C

Named Pipe (FIFO)

气泡启动时创建一个 named pipe，Hook 通过写入 pipe 发送事件。气泡端持续读取 pipe，天然有序，无竞态。

Pros

  • 实时推送，零轮询
  • 消息天然有序
  • 写入方不需要知道读取方状态

Cons

  • pipe 是单向的，重启气泡需要重建
  • Hook 写入 pipe 时如果读取端崩溃会阻塞
  • macOS 对 FIFO 的支持不如 Linux 稳定
